# Руководство пользователя TransitOS

Система автоматизированного учёта услуг АТП: заявки на перевозку, график автопарка, путевые листы, контроль топлива, отчёты.

Доступ: **десктоп** (Wails) или **браузер** (веб-сервер). Интерфейс адаптирован для ПК и смартфона водителя.

---

## 1. Вход в систему

1. Запустите приложение или откройте сайт (например http://localhost:8080).
2. На экране **TransitOS** введите логин и пароль.
3. Нажмите **Подключиться**.

При ошибке проверьте раскладку и демо-учётные записи (таблица ниже).

| Логин | Пароль | Роль | Назначение |
|-------|--------|------|------------|
| admin | admin123 | Администратор | Полный доступ, справочники, отчёты |
| dispatcher | disp123 | Диспетчер | Заявки, график, клиенты |
| driver1 | drv123 | Водитель | Рейсы и закрытие смены |

**Выход:** кнопка питания в правом верхнем углу.

---

## 2. Навигация

После входа сверху отображается **горизонтальная панель разделов** (dock):

- У **диспетчера и админа:** Обзор, Рейсы, Создать, Парк (график), …
- У **водителя:** Рейсы и Смена (на телефоне — нижняя панель).

Заголовок раздела и ФИО оператора — в шапке.

## 3. Роли и возможности

### 3.1 Администратор

| Раздел | Действия |
|--------|----------|
| Обзор | KPI: заявки за день, активные ТС, выручка, перерасходы топлива |
| Рейсы | Просмотр всех заявок (лента с маршрутом и статусом) |
| Создать | Новая заявка с автоматическим расчётом цены |
| Парк | График занятости ТС на сегодня (горизонтальные полосы рейсов) |
| Тарифы | Просмотр и добавление тарифов |
| ТС | Реестр автопарка, добавление машин |
| **Сотрудники** | Добавление и удаление учётных записей (водители, диспетчеры, администраторы) |
| Отчёты | Выручка, перерасходы, рейтинг водителей |

Дополнительно через API: журнал аудита (см. [developer/api-reference.md](developer/api-reference.md)).

Подробнее: [user-guide/administrator.md](user-guide/administrator.md)

### 3.2 Диспетчер

| Раздел | Действия |
|--------|----------|
| Обзор | Сводка за день |
| Рейсы | Все заявки |
| Создать | Оформление заявки (клиент, тариф, маршрут, ТС, водитель) |
| Парк | График — кто занят, кто свободен |

Не доступно: изменение тарифов/ТС, отчёты, пользователи.

Подробнее: [user-guide/dispatcher.md](user-guide/dispatcher.md)

### 3.3 Водитель

| Раздел | Действия |
|--------|----------|
| Рейсы | Заявки **на сегодня**, назначенные этому водителю |
| | **Старт рейса** → статус «В пути» |
| | **Завершить** → статус «Завершена» |
| Смена | Закрытие смены: пробег, топливо, комментарий |

**Контроль топлива:** если фактический расход более чем на **5%** выше нормы, без комментария смена не закроется.

Подробнее: [user-guide/driver.md](user-guide/driver.md)

## 4. Типовые сценарии

### 4.1 Создание заявки (диспетчер)

1. Раздел **Создать**.
2. Выберите клиента, тариф, адреса «откуда» / «куда», километраж.
3. При необходимости укажите простой (часы) и срочность.
4. Назначьте автомобиль и водителя.
5. Поле **Расчёт** покажет цену автоматически.
6. **Создать заявку** — заявка появится в **Рейсах** со статусом «Назначена».

**Формула цены:**  
`цена = (подача + км × тариф_км + простой_ч × тариф_ч) × коэффициент_срочности`

### 4.2 График парка

1. Раздел **Парк**.
2. Каждая строка — госномер; полосы — рейсы на сегодня.
3. Пустая полоса «свободен» — машина без заявок на текущую дату.

### 4.3 Рабочий день водителя

1. Войти как `driver1` / `drv123`.
2. **Рейсы** → **Старт рейса** по назначенной заявке.
3. По прибытии → **Завершить**.
4. В конце дня → **Смена** (или кнопка внизу на экране рейсов): ввести одометр и топливо.
5. При перерасходе >5% — обязательный комментарий.

## 5. Примеры API-запросов (веб-режим)

Токен получают при логине и передают в заголовке:

```http
Authorization: Bearer <token>
```

### Вход

```bash
curl -s -X POST http://localhost:8080/api/login \
  -H "Content-Type: application/json" \
  -d '{"login":"dispatcher","password":"disp123"}'
```

Ответ:

```json
{
  "token": "a1b2c3...",
  "user": { "id": "...", "login": "dispatcher", "role": "dispatcher", ... }
}
```

### Список заявок

```bash
curl -s http://localhost:8080/api/orders \
  -H "Authorization: Bearer TOKEN"
```

### Создание заявки

```bash
curl -s -X POST http://localhost:8080/api/orders \
  -H "Authorization: Bearer TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "clientId": "UUID-клиента",
    "vehicleId": "UUID-ТС",
    "driverId": "UUID-водителя",
    "fromAddr": "Склад №1",
    "toAddr": "ул. Ленина, 10",
    "distanceKm": 15,
    "idleHours": 0,
    "urgent": false,
    "tariffId": "UUID-тарифа",
    "scheduledAt": "2026-05-17T14:00:00Z"
  }'
```

### График ТС

```bash
curl -s http://localhost:8080/api/schedule \
  -H "Authorization: Bearer TOKEN"
```

Полный список: [examples/api.http](examples/api.http), [developer/api-reference.md](developer/api-reference.md)

## 6. Как данные сохраняются в базе

Приложение использует **LevelDB** (встроенное key-value хранилище, файлы на диске).

| Что | Где на диске | Ключ в БД (пример) |
|-----|--------------|-------------------|
| Каталог данных | `./data` или `ATP_DATA_DIR` | — |
| Файлы LevelDB | `{dataDir}/data/` | — |
| Пользователь | JSON | `user:{uuid}` |
| Индекс логина | ID пользователя | `user:login:dispatcher` |
| Сессия | JSON, TTL 24 ч | `session:{token}` |
| Заявка | JSON | `order:{uuid}` |
| Клиент, ТС, тариф | JSON | `client:`, `vehicle:`, `tariff:` |
| Путевой лист | JSON | `waybill:{uuid}` |
| Аудит | JSON | `audit:{uuid}` |

При каждом сохранении заявки сервис записывает запись в журнал аудита.

Подробнее с примерами полей JSON: [examples/storage-leveldb.md](examples/storage-leveldb.md)

## 7. Решение проблем

| Симптом | Действие |
|---------|----------|
| «Каталог данных занят» | Закройте второй экземпляр; удалите `data/.atp.lock` и `data/data/LOCK` |
| Не входит под демо-логином | Удалите `data/` и перезапустите (пересоздастся seed) |
| Пустой график | Убедитесь, что заявки с `scheduledAt` на **сегодня** |
| Водитель не видит рейсы | Заявка должна быть на него и на сегодняшнюю дату |
